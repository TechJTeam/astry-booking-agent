# Calling the astry-booking-agent API

The agent runs on FastAPI (the `vanna` framework), default port `8200`. This service is **not** a
typical REST CRUD API — there is essentially one chat endpoint, and every operation (looking up
services, checking availability, creating/rescheduling/cancelling appointments...) goes through
natural-language conversation, with the agent picking the right tool internally.

## 1. Auth — 2 required layers

| Header | Purpose | Set by |
|---|---|---|
| `X-API-Key: <AGENT_API_KEY>` | Service-to-service auth (only the POS frontend/BFF is allowed to call the agent) | The caller (POS frontend/BFF) |
| `Authorization: Bearer <JWT>` | The logged-in **Staff** member's JWT, forwarded as-is from the POS frontend | The caller, forwarding the Staff's existing JWT |

Details:

- `X-API-Key` is checked by `ApiKeyMiddleware` ([src/auth.py](../src/auth.py)) on **every route**
  except `/health`, `/docs`, `/redoc`, `/openapi.json`, `/`. If the server doesn't have
  `AGENT_API_KEY` set (dev), the middleware lets requests through — **production must set it**.
  Missing/wrong key → `401` JSON:
  ```json
  { "detail": "Unauthorized: thiếu hoặc sai X-API-Key." }
  ```
- `Authorization: Bearer <JWT>` is **not signature-verified by the agent** — the agent only
  decodes the payload to read `sub`/`email`/`salon_id`/`realm_access.roles`, then replays the
  **exact same** JWT when calling `astry-pos-be` (which is where RS256/JWKS verification actually
  happens). So the JWT must be a real, unexpired token issued by Keycloak to the actual Staff
  member performing the action — not a dedicated service account for the agent.
- Missing `Authorization` header → the agent does **not** return HTTP `401`. Since this is an SSE
  endpoint, the `MissingJwtError` is caught by the `vanna` framework and surfaced as an **error
  event inside the stream** (see section 3.3), while the HTTP status stays `200`. Clients must
  check for `type: "error"` in the stream themselves — they cannot rely on the status code.

## 1.1 Swagger UI

Auto-generated by FastAPI, no extra config needed. `ApiKeyMiddleware` already whitelists these
paths, so they're viewable without `X-API-Key`:

- Swagger UI: `http://localhost:8200/docs`
- ReDoc: `http://localhost:8200/redoc`
- OpenAPI JSON: `http://localhost:8200/openapi.json`

Only the real routes show up: `POST /api/chat`, `POST /api/chat_poll`, `GET /health` (the
WebSocket route isn't represented in OpenAPI). "Try it out" does issue a real request, but since
`chat` returns SSE, the response shows up as one raw `data: {...}` blob instead of a live
stream — good enough for a quick header/body sanity check, not a place to build a real chat UI.

## 3. `POST /api/chat` — main endpoint (Server-Sent Events)

### 3.1 Request

```
POST /api/chat
Content-Type: application/json
X-API-Key: <AGENT_API_KEY>
Authorization: Bearer <Staff JWT>
```

Body:

```json
{
  "message": "Book a haircut for Lan tomorrow at 3pm",
  "conversation_id": "c9f1b0a2-... (optional — omit on the first message, the agent generates one and returns it in the response)"
}
```

| Field | Required | Notes |
|---|---|---|
| `message` | ✔ | Natural-language message from the Staff member |
| `conversation_id` | ✘ | Omit on the first message of a conversation; subsequent turns **must** pass back the id received earlier so the agent keeps context (critical for the confirm-gate pattern in section 5) |
| `request_id` | ✘ | For tracing, auto-generated if omitted |

### 3.2 Response — SSE stream

The response is `text/event-stream`, each event shaped like:

```
data: {"rich": {...}, "simple": {...}, "conversation_id": "...", "request_id": "...", "timestamp": 1737...}

```

Terminated by:

```
data: [DONE]

```

- `rich`: full payload for a rich UI (component data, could be streamed text, tables, confirm
  buttons...).
- `simple`: a slimmed-down version (usually just text) for basic UIs — can be `null`.
- Clients can just read `simple.text` (if present) or aggregate text from `rich` to render a
  plain chat bubble.

### 3.3 Errors during processing (including a missing/invalid JWT)

Verified by testing against a live server — this differs from what you'd guess just from reading
the route code, because `vanna`'s `Agent.send_message()` wraps the entire per-message pipeline
(including user resolution, i.e. reading the `Authorization` header) in its own try/except and
**never lets the exception reach the SSE route handler**. So a missing/invalid Staff JWT does
**not** produce a distinct `{"type": "error", ...}` SSE event — it produces a **normal-looking
chunk sequence that still ends with `[DONE]`**, HTTP status `200`:

```
data: {"rich": {"id": "...", "type": "status_card", "data": {"title": "Error Processing Message", "status": "error", "description": "An unexpected error occurred while processing your message. Please try again.\n\nConversation ID: conv_...", "icon": "⚠️"}}, "simple": {"type": "text", "text": "Error: An unexpected error occurred. Please try again. (Conversation ID: conv_...)"}, "conversation_id": "conv_...", "request_id": "..."}

data: {"rich": {"id": "vanna-status-bar", "type": "status_bar_update", "data": {"status": "error", "message": "Error occurred", "detail": "An unexpected error occurred while processing your message"}}, "simple": null, "conversation_id": "conv_...", "request_id": "..."}

data: {"rich": {"id": "vanna-chat-input", "type": "chat_input_update", "data": {"placeholder": "Try again...", "disabled": false}}, "simple": null, "conversation_id": "conv_...", "request_id": "..."}

data: [DONE]
```

Important consequences:
- **The message shown to the client is always this generic text**, regardless of the actual cause
  (missing JWT, expired JWT, a tool crashing, an LLM API error...) — the specific reason (e.g.
  "Missing Authorization: Bearer \<JWT\>") is only written to the **server's own logs**
  (`logger.error(...)` inside `vanna`, plus this project's `traceback.print_exc()`), never sent to
  the client. Don't rely on `simple.text`/`rich` content to distinguish error causes on the client
  side — if you need that, check the server logs instead, or add your own client-side JWT expiry
  check before calling (see section 7).
- **Detecting an error client-side**: check for a `rich.type === "status_card"` chunk with
  `rich.data.status === "error"` (or `simple.text` starting with `"Error:"`). There is no
  top-level `"type": "error"` field to check — that would only appear if something failed
  *outside* `Agent.send_message` itself (e.g. serializing a chunk), which is rare in practice; if
  it does happen, the stream closes **without** `[DONE]`, so as a general rule clients should
  treat "connection closed with no `[DONE]` chunk" as an error too, not just look for a specific
  chunk shape.

### 3.4 `curl` example

```bash
curl -N -X POST http://localhost:8200/api/chat \
  -H "Content-Type: application/json" \
  -H "X-API-Key: dev-astry-booking-to-db-agent-key" \
  -H "Authorization: Bearer $STAFF_JWT" \
  -d '{"message": "Book a haircut for Lan tomorrow at 3pm"}'
```

(`-N` disables curl buffering, so you see the stream arrive in real time.)

### 3.5 JavaScript example (fetch + ReadableStream)

`EventSource` doesn't support custom headers (`X-API-Key`, `Authorization`), so use `fetch` and
read the stream manually instead of `EventSource`:

```js
const res = await fetch('http://localhost:8200/api/chat', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'X-API-Key': AGENT_API_KEY,
    'Authorization': `Bearer ${staffJwt}`,
  },
  body: JSON.stringify({ message, conversation_id: conversationId }),
});

const reader = res.body.getReader();
const decoder = new TextDecoder();
let buffer = '';

while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  buffer += decoder.decode(value, { stream: true });

  const lines = buffer.split('\n\n');
  buffer = lines.pop(); // incomplete trailing event, keep it for the next read

  for (const line of lines) {
    if (!line.startsWith('data: ')) continue;
    const payload = line.slice('data: '.length);
    if (payload === '[DONE]') return;

    const chunk = JSON.parse(payload);
    if (chunk.type === 'error') {
      console.error('Agent error:', chunk.data.message);
      return;
    }
    conversationId = chunk.conversation_id; // remember for the next chat turn
    // render chunk.simple?.text or chunk.rich depending on your UI
  }
}
```

## 4. Alternative endpoints (not the main one, but same `vanna` framework)

| Endpoint | Method | Notes |
|---|---|---|
| `/api/chat_poll` | POST | Same body as `chat`, but returns one full JSON response instead of streaming — use when the client can't easily handle SSE. Worse UX latency since it waits for the agent to finish completely. |
| `/api/chat_websocket` | WS | Equivalent to `chat` over WebSocket, send JSON `{"message": "...", "conversation_id": "..."}` per turn. |

Body/auth are identical to `chat` (`chat_poll` accepts `X-API-Key` + `Authorization` headers
normally; `chat_websocket` receives cookies/headers at handshake time).

## 5. Confirm-gate pattern — every WRITE operation (create/reschedule/cancel appointments, join waitlist)

The agent **never writes data on the first call**. Every write tool (`create_appointment`,
`reschedule_appointment`, `cancel_appointment`, `confirm_or_decline_pending_appointment`,
`join_waitlist`) is a 2-step flow, entirely inside the natural conversation — **the client doesn't
need to know which tool is running**, it just needs to forward the right `conversation_id` between
turns:

1. **Turn 1** — Staff types a request (e.g. "book a haircut for Lan at 3pm tomorrow"). The agent
   internally calls the tool with `confirm=false`, does NOT call any write API on
   `astry-pos-be`, and just replies with text asking for confirmation (e.g. "Confirm booking:
   customer Lan, 3:00pm tomorrow, Haircut service. Correct?").
2. **Turn 2** — Staff replies affirmatively (e.g. "yes", "correct", "confirm") — the client sends
   this message with the **same `conversation_id`** from turn 1. The agent recognizes the
   confirmation intent and calls the tool again with `confirm=true`, and only now actually
   `POST`s/`PATCH`es `astry-pos-be`.

→ For integrating clients: just a normal chat flow (send message, receive reply, forward
`conversation_id`) — no need to build a separate confirmation UI, the agent asks for confirmation
in natural language itself.

## 6. Full example flow (2 HTTP `chat` turns)

```bash
# Turn 1 — no conversation_id
curl -N -X POST http://localhost:8200/api/chat \
  -H "Content-Type: application/json" -H "X-API-Key: $AGENT_API_KEY" \
  -H "Authorization: Bearer $STAFF_JWT" \
  -d '{"message": "Book a haircut for Lan tomorrow at 3pm"}'
# -> agent asks for confirmation, returns conversation_id="abc-123" in every chunk

# Turn 2 — must pass back the conversation_id received above
curl -N -X POST http://localhost:8200/api/chat \
  -H "Content-Type: application/json" -H "X-API-Key: $AGENT_API_KEY" \
  -H "Authorization: Bearer $STAFF_JWT" \
  -d '{"message": "Confirm", "conversation_id": "abc-123"}'
# -> agent actually calls POST /appointments on astry-pos-be, replies with the booking result
```

## 7. Integration notes

- **There's no separate REST endpoint per business action** (no `POST
  /agent/create-appointment`-style route) — everything goes through `chat`/`chat_poll`/
  `chat_websocket` in natural language. To make the agent do something, phrase it as a `message`.
- **`conversation_id` must be remembered client-side** across turns of the same conversation — the
  agent has no other way to link context together.
- If the Staff's JWT expires mid-conversation, the verification failure happens on the
  `astry-pos-be` side when the agent calls back into it (usually surfaced as a tool returning
  `error: pos_api_error` with a 401 message in the reply body, not as an HTTP 401 at the
  `chat` layer). Clients should check JWT expiry themselves before calling.
- By default (`CONVERSATION_STORE_URL` unset), conversation history is stored **in-RAM** and lost
  on agent restart — clients shouldn't treat `conversation_id` as durable across deploys unless
  that variable is set (see [README.md](../README.md), "Conversation store" section).
